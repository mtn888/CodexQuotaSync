# Codex Quota Sync 方案决策记录

## 已确认范围

- 主 Windows 作为唯一 collector，复用现有 Codex 登录态，不新增登录流程。
- 其他 Windows 使用同一悬浮组件的 viewer 模式；Android 使用原生桌面小组件。
- 同步服务部署为 UnRaid 上的 Go 单容器，只保存最新 `state.json`，不使用数据库。
- 服务仅提供 HTTP 和自定义端口；公开读取，HMAC 鉴权写入。
- Windows 展示和活动同步每分钟一次；额度常规缓存 5 分钟，临近重置 15 分钟内缓存 1 分钟。
- Android 使用 WorkManager 每 15 分钟自动更新、网络失败退避重试，并支持系统 AppWidget 备用触发和点击刷新。
- 同步 5 小时/周额度、各自重置时间、执行中、待审批和待用户输入数量；界面把后两项合并为“待处理”。
- 开发阶段交付未签名 Windows 安装包和 Android Debug APK。

## 关键取舍

选择 UnRaid 单容器而不是飞书共享文档：共享文档的自动写入仍需要应用凭据或 OAuth，并受限流、缓存和文档格式影响；已有 NAS 时，单 Go 进程与单 JSON 文件更容易部署、备份和排障。

活动数量采用 Codex Hooks 的 best-effort 统计。它覆盖本机顶层任务，不保证所有 Codex 版本都会为内置 `request_user_input` 发出 Pre/PostToolUse，因此“待用户输入”可能暂时仍显示为“执行中”；不统计 subagent、CLI、云任务或排队 follow-up。

按需求使用明文 HTTP。HMAC 只保护写入完整性和基本防重放，不加密传输内容。完整协议见 [架构与协议](架构与协议.md)，部署见 [UnRaid 部署说明](UnRaid部署说明.md)。
